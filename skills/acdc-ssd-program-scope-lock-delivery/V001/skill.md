# Objective

Define and deliver ACDC Skill Succession Developments (SSDs) as a standing, constitution-aligned ACDC program by:

- separating SSD ownership from any single project
- sequencing formal and informal SSDs according to organizational maturity
- assigning advisor, BoD, committee, and participant responsibilities
- specifying slides, assessments, certificates, and records requirements
- integrating BCC participation as learners and mentees without premature project execution responsibility

This skill exists to prevent SSD drift into vague workshops, project-specific dependency, or ungoverned programming.

# Trigger

Use this skill when a conversation, semester plan, or governance packet requires any of the following:

- defining SSD scope for a semester
- distinguishing formal SSDs from informal skill sessions
- aligning SSD delivery to the ACDC Constitution and Bylaws
- assigning advisor skill module obligations
- establishing certificate and quiz administration
- creating or resetting an SSD program after execution drift
- integrating BCC students into ACDC through SSD participation or transfer mentoring
- explaining that SSDs are an ACDC program and not a SSUNC-owned artifact

# Do Not Use When

Do not use this skill when:

- the request is only to summarize the ACDC Constitution
- the request is only to draft a single email with no reusable program logic
- the request is purely about SSUNC technical execution rather than ACDC program design
- the source material lacks governing authority and is only speculative brainstorming
- the user needs a one-off workshop outline unrelated to SSD governance

4. The requested governance artifact type or planning need

# Outputs

- A constitution-aligned SSD scope lock for the semester
- Delivery model defining formal vs informal SSDs
- Role assignment for BoD, advisors, committee, and participants
- Slides / briefing requirements for each formal SSD
- Assessment and certificate rules
- DRAM records and minutes requirements
- BCC SSD participation and transfer mentoring boundaries
- Optional BoD motion language for temporary sequencing adjustment

# Support Layers

- Constitution layer: Article II §2.1, Article IV §§4.4–4.8, Article XIII §13.4, Bylaw 5.1, Bylaw 10.1, Bylaw 14.4, and Bylaw 16.3 govern the program baseline.
- Program layer: SSDs are treated as ACDC Programs under the 4Ps, not as SSUNC deliverables.
- Committee layer: the Learning & Skill Development Committee designs and maintains instructional modules, quizzes, and certifications.
- Records layer: DRAM records appointments, completions, certificates, votes, and any semester sequencing adjustments in official minutes.
- Delivery layer: project artifacts may be used as case studies, but projects do not own SSDs.
- Mentoring layer: BCC participation is educational and preparatory, not executional.

# Procedure

1. Confirm program identity.
   State explicitly that SSDs are an ACDC-wide standing program, not a SSUNC-owned workstream.

2. Load constitutional authority.
   Identify the controlling clauses:
   - Article II §2.1 for Programs within the 4Ps
   - Article IV §4.4 and Bylaw 5.1 for RAM advancement via SSD completion
   - Bylaw 10.1 for advisor skill module obligations
   - Article XIII §13.4 for Learning & Skill Development Committee ownership
   - Bylaw 14.4 for Transfer Mentor as a functional role
   - Bylaw 16.3 for Programs requiring assessment and certificate completion

3. Define semester SSD inventory.
   List all candidate SSDs for the semester and classify each into one of two categories:
   - formal SSD
   - informal skill session

4. Apply sequencing discipline.
   Select only the SSDs that the organization can realistically run with instructional integrity.
   Formal SSDs must be limited to those the BoD and committee can support with slides, assessment, certificate workflow, and records control.

5. Specify formal SSD delivery requirements.
   For each formal SSD, define:
   - owner
   - session length
   - delivery format
   - slide or written briefing requirement
   - assessment method
   - certificate issuance criteria
   - archive requirements

6. Specify informal session boundaries.
   For each informal session, define:
   - purpose
   - whether demonstration or hands-on only
   - no certificate status unless explicitly upgraded later
   - no RAM advancement credit unless constitutionally approved and documented

7. Assign organizational roles.
   - BoD: approves sequencing, owners, and scope lock
   - Learning & Skill Development Committee: prepares materials, quizzes, and certification flow
   - Advisors: deliver or review one skill module when scheduled and aligned to expertise
   - DRAM: records completion, votes, appointment, and certificate status
   - ERAM/RAM support: logistics, facilitation, grading support if authorized

8. Define assessment and certificate mechanics.
   Formal SSDs must include:
   - 60–90 minute instructional delivery or equivalent approved format
   - slides or briefing document tied to real artifacts
   - a short quiz or written exercise
   - defined completion threshold
   - certificate issuance recorded by DRAM
   - advisor and corresponding chair signature when operating as a Program under Bylaw 16.3

9. Integrate real examples.
   Use actual ACDC / SSUNC artifacts as examples without making SSDs dependent on the project. Typical mappings:
   - Requirements SSD → brutal MVP, scope lock, verification language
   - Trade Studies SSD → constrained decision matrix, rationale under resource limits
   - Carpentry Session → fixtures, mounting, enclosure discipline
   - 3D Printing Session → tolerances, fit checks, print iteration, manufacturability

10. Establish BCC participation model.
    Allow BCC students to:
    - attend SSDs
    - complete assessments
    - earn certificates where permitted
    - receive preparatory mentoring
    Do not allow pre-transfer project execution ownership, demo-critical access, or committee authority.

11. Establish BCC mentoring model.
    Define transfer mentoring as informational and preparatory. Covered areas may include:
    - NJIT academic expectations
    - course sequencing
    - workload realism
    - portfolio and resume guidance
    - ACDC structure after transfer
    Exclude:
    - project task assignment
    - subsystem ownership
    - deadlines tied to ACDC execution

12. Write the semester scope lock.
    Produce the semester SSD statement with:
    - formal SSD list
    - informal SSD list
    - advisor role
    - certificate and quiz administration method
    - BCC participation and mentoring rules
    - any time-boxed sequencing adjustment if needed

13. Record governance actions.
    If sequencing differs from the full ideal SSD catalog, generate BoD motion language clarifying that:
    - SSD authority is preserved
    - only semester sequencing is adjusted
    - the change is temporary
    - expiration occurs automatically at semester end unless renewed

# Decision Logic

| Condition | Action |
|----------|--------|
| SSDs are being discussed as part of ACDC capability-building | Treat as ACDC Program design |
| SSDs are being described as part of SSUNC ownership | Correct the framing and separate program from project |
| Constitution requires module, assessment, and certificate integrity | Keep SSD formal only if these can actually be administered |
| semester execution load is high | reduce formal SSD count and move remaining modules to informal sessions |
| advisor participation exists but scheduling is weak | assign advisor module to a bounded, domain-fit SSD with DRAM-tracked scheduling |
| BCC involvement is requested before transfer | route through SSD attendance and transfer mentoring only |
| project artifacts are available | use them as case studies, not as governing anchors |
| no assessment or certificate path can be supported | downgrade the SSD to an informal session |

# Validation

A correct execution of this skill must ensure all of the following:

- SSDs are explicitly identified as an ACDC program
- at least one governing constitutional basis is cited or embedded in the plan
- formal vs informal SSDs are clearly distinguished
- formal SSDs include delivery, assessment, and certificate requirements
- advisor participation is bounded and role-correct
- DRAM records responsibility is included
- BCC mentoring is separated from project execution
- project examples support the SSDs but do not own them
- any sequencing adjustment is framed as temporary operational control, not constitutional repeal

# Rules

- Never frame SSDs as owned by SSUNC.
- Never allow projects to define SSD existence.
- Do not label an SSD formal unless slides or written instructional content, an assessment, and certificate mechanics are specified.
- Do not assign BCC students pre-transfer execution responsibility on ACDC projects.
- Do not convert an informal session into RAM-credit-bearing work without constitutionally valid assessment and documentation.
- Keep semester scope realistic; sequencing is allowed, abandonment is not.
- DRAM must be included wherever completion, vote, appointment, or certification status is involved.
- Advisor obligations must be scheduled and bounded; do not leave them implied.
- Use real artifacts to teach skills, but do not let the teaching artifact redefine the program.

# Failure Modes

| Failure | Recovery |
|--------|----------|
| SSDs are treated like vague workshops | reclassify each module as formal or informal and add delivery mechanics |
| SSDs are described as SSUNC work | restate ACDC program ownership and move SSUNC to example-only status |
| Advisors are named but not scheduled | assign module owner, date window, and DRAM recording requirement |
| Certificates are promised without assessment | downgrade to informal session or add a valid assessment |
| Too many formal SSDs are planned | reduce formal count and preserve the remainder as informal sessions |
| BCC students are pulled into execution too early | reroute them to SSD attendance and transfer mentoring only |
| RAM advancement language is used without certificate flow | add completion threshold, signatures, and records path |
| semester adjustment is mistaken for constitutional change | issue explicit BoD sequencing language with automatic expiration |

# Dependencies

- ACDC Constitution and Bylaws
- Current semester SSD inventory
- Learning & Skill Development Committee support
- DRAM records workflow
- Advisor availability
- Meeting minutes or governance record system
- Optional project artifacts for case-study teaching

# Assumptions

- The organization wants SSDs to remain active but realistically sequenced.
- The semester may not support full formal execution of every SSD.
- Formal SSDs must be credible enough to support membership advancement pathways.
- BCC involvement is valuable as preparation and continuity, not as premature execution.
- ACDC benefits from separating skill production from project consumption.
