# Objective

Execute a minimal, high-signal pilot showcase that evaluates AI and autonomous systems projects using structured intake, selective cohort formation, and faculty-based evaluation, while avoiding overengineering and maintaining execution velocity.

---

# Trigger

Use this skill when:

- launching a new technical showcase or competition pilot
- coordinating projects, judges, and outreach across institutions
- needing a minimal but credible evaluation system
- transitioning from prototype/system demo → program-level execution
- building a founding cohort for a recurring technical platform

---

# Do Not Use When

Do not use this skill when:

- the goal is a large-scale public event
- there is no working technical anchor system
- the effort is purely informational or academic
- no selection or evaluation mechanism is required
- the system lacks defined execution ownership

---

# Required Inputs

1. Core technical anchor (e.g., validated system like ASVP/SSUNC)
2. Target event window (date or timeframe)
3. Access to participants (students, labs, or teams)
4. Ability to reach faculty or evaluators
5. Physical or virtual execution space

---

# Optional Inputs

- Partner institutions
- Advisor or faculty alignment
- Existing outreach channels (Whova, email lists, etc.)
- Prior demo artifacts (videos, logs)
- Institutional affiliation leverage

---

# Outputs

- Participant intake system (single form)
- Judge confirmation system (single form)
- Defined evaluation rubric
- Curated founding cohort (3–5 teams)
- Confirmed judge panel (2–3 minimum)
- Executable event structure (2–3 hour format)
- External outreach pipeline
- Follow-on expansion pathway

---

# Support Layers

- Intake Layer: participant filtering and classification
- Judge Layer: evaluation credibility system
- Cohort Layer: selected high-signal builders
- Execution Layer: physical or structured event
- Outreach Layer: institutional expansion
- Validation Layer: real-system demonstration and scoring

---

# Procedure

1. Define Scope Lock
   - limit to pilot scale (≤10 teams)
   - enforce real systems or structured concepts only

2. Create Intake Form
   - collect name, affiliation, project, lane, description, stage
   - include commitment filter

3. Create Judge Form
   - collect confirmation, expertise, availability, conflicts

4. Define Lanes
   - Demo Lane: real system with observable behavior
   - Concept Lane: architecture + validation plan

5. Build Evaluation Rubric
   - originality
   - usefulness
   - system design
   - validation
   - IP/product potential

6. Execute Outreach
   - targeted faculty invitations
   - targeted student/team recruitment
   - prioritize quality over quantity

7. Form Founding Cohort
   - select 3–5 strongest teams
   - reject or defer low-signal entries

8. Confirm Judges
   - secure 2–3 faculty minimum
   - ensure alignment with rubric

9. Lock Event Structure
   - 5 min intro
   - 60–90 min showcase
   - 15–20 min judging
   - 10 min awards

10. Execute Event
   - enforce safety and demonstration rules
   - prioritize real behavior over explanation

11. Capture Outputs
   - projects
   - evaluations
   - judge feedback
   - future candidates

---

# Decision Logic

| Condition | Action |
|----------|--------|
| Too many participants | reduce to top 3–5 |
| Weak projects | delay or refine cohort |
| No external teams | proceed with internal pilot |
| No judges confirmed | halt event execution |
| Demo unsafe | allow recorded evidence |
| Concept vague | reject entry |

---

# Validation

The system is valid if:

- at least 3 strong projects are present
- at least 2 judges are confirmed
- event runs within defined time structure
- real systems or credible concepts are demonstrated
- evaluation is completed using rubric

Success is defined as:
- execution without collapse
- credible evaluation occurs
- cohort can be reused or expanded

---

# Rules

- keep system minimal (no overengineering)
- do not exceed 10 total projects
- prioritize execution over planning
- enforce real systems or structured concepts
- judges must not be optional
- no expansion before pilot success

---

# Failure Modes

| Failure | Cause | Recovery |
|--------|------|----------|
| No participation | weak outreach | target specific builders |
| Judge drop-off | unclear role | reinforce low commitment |
| Event chaos | too many teams | reduce scope |
| Weak credibility | no structure | enforce rubric |
| Overload on lead | too many systems | reduce tooling |

---

# Dependencies

- Google Forms (or equivalent)
- Communication channel (email/DM)
- Physical or virtual event space
- Basic documentation (rubric)

---

# Assumptions

- small team (≤4 people)
- limited time and resources
- pilot-first strategy
- technical anchor exists
- founder is load-bearing operator
