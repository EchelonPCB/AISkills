# Objective

Transform a completed retrospective synthesis into a **high-impact delivery format** that:

- communicates system evolution clearly
- enforces structure and hierarchy
- separates execution from optics
- highlights real engineering progress
- includes a “brutal truth” layer

This is the **presentation layer** of retrospective synthesis.

---

# Trigger

Use this skill when:

- retrospective synthesis is complete
- user wants to “see what it actually looked like”
- communication clarity matters (stakeholders, self-evaluation)
- output must be high-signal, structured, and persuasive

Common activation phrases:
- what was my month like
- show me the breakdown
- summarize my progress (but structured)
- retrospective output
- brutal analysis
- executive reflection

---

# Do Not Use When

- synthesis has not been performed yet → use period-retrospective-synthesis
- single artifact analysis → reference
- user wants raw data → do not interpret
- casual summary requested → lightweight output instead

---

# Required Inputs

1. Completed retrospective synthesis (implicit or explicit)
2. Supporting artifacts (optional but preferred)

---

# Optional Inputs

- tone preference (formal, aggressive, neutral)
- audience (self, executive, external)
- emphasis (technical vs organizational)

---

# Outputs

A structured retrospective with:

1. 4–6 numbered thematic sections
2. Each section includes:
   - title
   - evidence-backed explanation
   - system interpretation
3. A **“What You Actually Did Well”** section
4. A **“Where You Were Weak”** section
5. A **Final Compression Statement**

---

# Support Layers

- Communication layer (clarity, persuasion)
- Engineering layer (technical accuracy)
- Strategic layer (positioning, leverage)
- Reality filter layer (removes fluff)

---

# Procedure

## Step 1 — Structure the Narrative

Create:

- 4–6 sections representing major themes
- Order them from:
  - strategic → technical → execution

---

## Step 2 — Enforce Section Format

Each section must:

1. Name the theme clearly  
2. Explain what happened  
3. Anchor to evidence (implicit or explicit)  
4. State why it matters  
5. Connect to system-level impact  

---

## Step 3 — Apply Language Compression

- Remove filler words
- Use short, declarative sentences
- Avoid academic tone
- Prefer:
  - “This did X”
  - “This enabled Y”

---

## Step 4 — Insert “Reality Layer”

Add:

### What You Did Well
- Only include real execution
- Must map to system progress

### Where You Were Weak
- Must include:
  - integration gaps
  - unresolved risks
  - overstatements
- No soft language

---

## Step 5 — Final Compression

Produce:

→ ONE sentence defining the period

Rules:
- must reflect transformation
- no fluff
- no adjectives unless necessary

---

# Decision Logic

| Condition | Action |
|----------|--------|
| multiple artifacts support claim | treat as validated |
| only planning present | downgrade importance |
| messaging strong but execution weak | classify as optics |
| execution + validation present | elevate to milestone |
| incomplete system | flag in weakness section |

---

# Validation

Output is valid if:

- 4–6 structured sections exist
- each section has system-level meaning
- includes “did well” section
- includes “weakness” section
- includes final compression line
- no chronological storytelling

---

# Rules

- DO NOT narrate events
- DO NOT inflate planning
- DO NOT hide weaknesses
- DO NOT mix execution with speculation
- ALWAYS separate:
  - build vs validate vs integrate

---

# Failure Modes

| Failure | Cause |
|--------|------|
| reads like essay | lack of structure |
| too positive | missing reality layer |
| vague | no system framing |
| confusing | poor section hierarchy |
| overhyped | optics not filtered |

---

# Dependencies

- period-retrospective-synthesis skill
- access to artifact context
- system-level reasoning ability

---

# Assumptions

- synthesis already occurred
- user prefers high-signal output
- clarity > verbosity
